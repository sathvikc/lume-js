**Experimental `src/` edits currently applied on this branch:** `src/core/state.js` carries two `// EXPERIMENT:` markers. (1) The Round-6 **no-subscriber fast-path** — the `set` trap skips all notify/flush work when a store has zero listeners and no beforeFlush hooks; behaviourally transparent and a graduation candidate. (2) The Round-11 **`__setDirtySink`** opt-in change-signal (an exported setter + one null-checked `sink.add` inside the fast path) that feeds `mode=dirty`; also behaviourally transparent (null by default). **All 488 tests still pass with both.** Only `dirty` consumes the sink; if the fast-path graduates to `main`, drop `__setDirtySink` (Round 11 found the dirty list is not a responsiveness lever). To measure *without* these, `git stash -- src/core/state.js` and re-run. Nothing else in `src/` is patched — the Round-12 worker pipeline (`worker`/`whybrid` + `worker-paint.js`) is entirely lab-side and needs **no** kernel change (that is part of its finding: the reactive half integrates as-is).

## The question

Can Lume keep the UI responsive (input smooth, pixels current) when thousands of reactive cells update per frame under heavy CPU load (DevTools 20× throttle)? The frozen `feature/render-queue` addon (`renderQueue`) was the first attempt; the maintainer wants the whole direction stress-tested and, where possible, improved or reinvented.

```bash
npm run dev -- --port 5199 --strictPort           # serve examples (the lab reloads via HMR on edit)
PLAYWRIGHT_SKIP_BROWSER_DOWNLOAD=1 npm i --no-save playwright-core   # research-only, not a project dep
```

## Modes implemented in the lab (URL `?mode=`)

| mode | what it does | verdict |
|---|---|---|
| `off` | baseline: one `effect()` per cell, DOM write synchronous in the flush | the bar |
| `rq` | the frozen `renderQueue` addon (tiers, budget, IO, input-preempt) | KILL — see below |
| `simple` | ~15-line rAF coalesce: mark dirty, one rAF applies ALL dirty | fair, no freeze, no budget |
| `adaptive` | synchronous applies until a janky frame, then defer | no better than `off` anywhere |
| `cv` | `off` + CSS `content-visibility:auto` | best for sparse/large grids (free) |
| `bounded` | round-robin CURSOR drain (the maintainer's idea) | fixes rq's starvation; fair; bounded staleness |
| `pull` | NO per-cell effects — one rAF sweeps cursor, reads current state, paints changed within budget | attacks the flush; best in storm |
| `smart` | `pull` + input-reserved dynamic budget (small while interacting, 5× when idle) | best input of the pull family |
| `sliced` | LAYER 3 — also time-slice the WRITE pass: producer buffers latest value, budgeted loop applies+paints a slice/frame | chunking works; kernel write path is the floor |
| `plain` | pull renderer over a PLAIN array — no `state()`, producer writes raw memory; scheduling identical to `pull` | the theoretical write floor (~0% kernel); confirms how little is left under the fast-path |
| `smartcv` | `smart` pull renderer on a `content-visibility:auto` grid (Layer 1 + Layer 2 stacked) | BEST responsiveness measured — the layers compose (see Round 7) |
| `canvas` | PAINT-FLOOR probe — reactive stores + byte-identical `pull` schedule, but paints changed cells as rects into ONE `<canvas>` (grid hidden), no per-cell DOM | diagnostic only, never shippable — the 16 ms floor (Round 8): DOM-per-cell is ~99% of remaining input delay |
| `vis` | `smartcv` + an `IntersectionObserver` visible-set — sweep/writes touch only visible cells; offscreen refresh lazily on reveal | NULL static (Round 9) and **WORSE under scroll** (Round 14): occlusion culls the cheap sweep, not the paint; under scroll the observer churn + O(N) list rebuilds + reveal-repaint make it +78% vs `smartcv`. Dead in every regime |
| `dirty` | `smartcv` + O(changed) drain of a kernel dirty-signal sink (`__setDirtySink`) instead of the O(N) sweep | NULL on input (Round 11): sweep was never the bottleneck; the `sink.add` even taxes the write path. Only win is an O(1) backlog (measurement aid) |
| `worker` | OFF-MAIN-THREAD pipeline — a dedicated worker owns produce+diff+paint into an `OffscreenCanvas` (`transferControlToOffscreen`); main thread does zero render work | BREAKS the paint floor (Round 12): input 5.8–10.5 ms @3000, 9.2 @8000 (vs smartcv 603–1576); main thread 77–93% free. Even software-throttled 20×, input stays ~5–7 ms — input decouples from render load. Diagnostic-shaped: no per-cell DOM, so out of the `data-*` model |
| `whybrid` | HYBRID — reactive `state()` on the MAIN thread; a smart-style loop diffs and posts changed cells to the worker to blit (paint-only off-thread) | 15–33× input over smartcv (21.5–47.7 ms) while KEEPING main-thread state (Round 12). Residual cost = main-thread diff+serialize+postMessage; pixels lag under extreme load (drop-stale). Proves the reactive half integrates with no core change |

URL knobs: `regime` (`storm`/`dashboard`/`burst`/`quiet`), `cells`, `budget`, `churn`, `dot` (0 removes the jank-dot render artifact — **always use dot=0 for real numbers**), `text` (0 = paint-only apply), `render` (0 = keep flush, skip DOM writes — isolates rendering cost), `wrate` (`worker` mode only: software slow-device throttle for the worker thread, 1 = native/free core — see Round-12's measurement note).

## What has been TRIED and FOUND (do not redo)

1. **The freeze is real and root-caused.** `renderQueue` drains ~16 cells/frame (2ms budget under 20×) while a storm re-dirties thousands; backlog pins, staleness grows unbounded. Budget tuning (2→32ms) does not fix it. *(FINDINGS "The freeze".)*
2. **Starvation bug in `renderQueue`.** Under uniform re-dirty order (a grid iterating cells 0..N — the common case) the Set drain re-adds every entry in the same order, so it repaints ~8 front cells up to 13× while **2984/3000 never paint**. The addon's no-starvation test only re-dirties one entry, so it ships green. `bounded`/`pull`/`simple` do not starve. *(FINDINGS Round 2; `round2.mjs starve`.)*
3. **renderQueue's input win is ~1.5× at best (median), never ≥5×**, and it freezes pixels under sustained load. `content-visibility` (free CSS) beats it 2–3× in realistic regimes. **Verdict: KILL** the shipped addon. Also 0.37 KB over the 8 KB all-in-one budget (budget kept at 8; the silent 8→9 raise was reverted). *(FINDINGS Rounds 1–2.)*
4. **Prototype's 7.4× vs my ~1.5×** = fixed-floor : spreadable-apply ratio. Headed (xvfb) did NOT change it — floor persists in both paths, so it is not a headless artifact. Residual caveat: both environments are ~5× slower natively than the prototype's machine, so on fast hardware the input benefit could be larger. Freeze/starvation are hardware-independent. *(FINDINGS Round 2/3; `round2.mjs fidelity`.)*
5. **fps ceiling.** Native, a full storm holds 60fps for only a few hundred cells; a dashboard ~1200. Rescheduling (pull/bounded/rq) does NOT raise the ceiling (same total work). Only **reducing work** raises it: `content-visibility` holds 60fps to ~2000 dashboard cells. *(FINDINGS Round 3; `headroom.mjs`.)*
6. **Main-thread free time.** Budgeted schedulers (pull/bounded) roughly HALVE the worst blocking task vs `off` — that (not fps) is what they buy. But nothing frees the thread from the WRITE pass. *(FINDINGS Round 3.)*
7. **`smart` (drop-stale + input-reserved budget)** had the best input of the pull family (burst: 1218ms vs 1486–1557ms fixed). Principle 2 works, but its ceiling is the write pass. *(FINDINGS Round 4; `smart.mjs`.)*
8. **Layer 3 (`sliced`): time-slicing the write pass.** Chunking WORKS at the JS level (producer 7ms, slice 16ms, budget-respected). But a CPU profile shows the real floor is the **Lume kernel write path** — `set` + `notifySubscribers` + `flushBatchedStates` ≈ **27% of active CPU**, O(N) in writes, even with ZERO subscribers. Chunking spreads it, doesn't reduce it; input at 3000 cells (~1155ms) ≈ pull/off. *(FINDINGS Round 5 / `results/layer3-summary.json`.)*
9. **Round 6 — the write-path floor is removed, two ways.** (a) **Kernel no-subscriber fast-path** (`src/core/state.js`, applied): the `set` trap skips notify/flush when a store has no listeners + no beforeFlush hooks. `pull` storm worst write-pass frame **156→16ms (~10×)**; profile kernel write path **~9%→~1.5%** (`notifySubscribers`/`takeEffects` vanish); **all 488 tests pass unchanged**. (b) **`plain` mode** (pull over a plain array, no `state()`): profile shows **~0% kernel write path** (97% `(program)`), `prod` ~8–16ms. The fast-path captures ~85% of the win *while keeping the reactive API*; plain buys only the last sliver by dropping reactivity. Neither raises the fps ceiling (still content-visibility) nor fixes input under a pure paint-bound storm — they remove the O(N) write floor Round 5 diagnosed. Use `prod` (worst write-pass frame), not input-delay (paint-dominated) or `worstBlock` (throttle-inflated). *(FINDINGS Round 6 / `results/round6-writepath-floor.json`, `exp1-fastpath.json`, `exp2-plain.json`.)*
10. **Round 8 — the paint floor is 16 ms; DOM-per-cell is ~99% of what's left, so scheduling headroom is exhausted.** The `canvas` probe (reactive stores + byte-identical `pull` schedule, but painting into one `<canvas>` instead of N DOM boxes) hits **median input 16 ms / 12.8 fps** at dashboard 3000/20× — vs `smartcv` **388 ms / 1.3 fps** (24× gap) and `off` **1977 ms / 0.3 fps** (89×). Decomposition in the same throttle band: `pull` (1420) ≈ `plain` (1378) → the reactive **write** is now ~3% of input (fast-path confirmed on the input axis); `pull` (1420) → `canvas` (16) → the **DOM paint** of individual boxes is the other ~99%. Consequence for the whole study: every scheduler moves *when* the N DOM writes happen but not *that* there are N of them, and Round 8 shows those writes are the cost — so `smartcv` is within a small constant of the best any scheduler can do inside the per-cell DOM model. **The only remaining large levers reduce the live DOM-box count** (occlusion write-culling → item 9; virtualization) or leave per-cell DOM (`canvas`/`OffscreenCanvas`, out of scope). Stop optimizing the schedule; reduce the boxes. *(FINDINGS Round 8 / `results/round8-item12-paintfloor.json`.)*
11. **Round 9 — occlusion write-culling (`vis`) is a NULL result on input; confirms Round 8.** `vis` = `smartcv` + an `IntersectionObserver` visible-set (sweep/writes touch only visible cells). Occlusion verifiably active (`visVisibleCount` 3556/8000, 4340/12000). But at dashboard 12000, two clean ~22–23× runs give `smartcv` 1847→2465 and `vis` 2508→2419 — **the direction flips, so input is indistinguishable**. The sweep it culls was never the bottleneck (writes ~3% of input, Round 8); the visible paint it can't cull is. `vis` even paints *more* visible boxes/frame (budget concentrated on-screen), a mild mechanism against it. Only genuine move: visible backlog lower (920–1365 vs 3061–3256) but denominator-confounded and absent from input/staleness. Occlusion does NOT raise the ceiling like `cv` did (cv culls paint, vis culls only the sweep). Its real use case — **scrolling** — was untested at Round 9; Round 14 tested it and `vis` came out **+78% worse than `smartcv`** (see item 16 / next-step (c)). *(FINDINGS Round 9 / `results/round9-vis-confirm.json`.)*
12. **Round 10 — per-cell text (layout) is a real SECONDARY lever, ~a quarter to a half of DOM cost.** Same modes with `text=1` (number+colour) vs `text=0` (colour only), dashboard 3000/20×: `off` 1935→1569 (−19%), `cv` 889→437 (**−51%**), `smartcv` 411→318 (−23%), `canvas` 26→17 (−34%) — and every `text=0` case ran at equal-or-harsher throttle, so the savings are *understated*. Dropping per-cell `textContent` (which forces layout/reflow) is shippable guidance: colour-only cells (CSS var/class, keep the number out of the churny path) are materially cheaper. **But text is not the floor** — even colour-only, `smartcv` (318) is still 19× above `canvas` (17); the dominant residual is the per-cell style-recalc + background paint of the DOM box itself, which only `canvas` removes. Untested adjacent lever: direct `style.backgroundColor` vs the inheriting `--h` custom property. *(FINDINGS Round 10 / `results/round10-text{0,1}.json`.)*
13. **Round 11 — O(changed) dirty pull (`dirty`) is NOT a responsiveness lever; confirms Round 8 a third time.** New kernel `EXPERIMENT`: opt-in `__setDirtySink(set)` adds each changed store to a sink inside the fast path (O(1), no notify — 488 tests still pass). `mode=dirty` = `smartcv` but drains the sink O(changed) instead of sweeping O(N). Result: **no consistent input win** — dashboard 3000 indistinguishable (449 vs 483), dashboard 12000 clean *worse* (1824 vs 2464), storm 3000 better-despite-harsher-throttle (458@17.9× vs 360@23.6×): all inside the throttle-drift noise. Input is paint-bound, and the sweep was never O(N)-per-frame anyway (budget breaks it after ~paints/churn scans). Two findings cut *against* it: the `sink.add` **taxes the write path** (`prod` 61 vs 42 at 12000, partly undoing the Round-6 fast-path win), and a `Set` is the wrong structure (its churn loses to the sweep's monomorphic array reads; a flat dirty-array would close the constant gap but not the structural verdict). Sole clean win: **O(1) backlog** (`dirtySink.size`) — a measurement aid for future convergence curves, not a perf lever. *(FINDINGS Round 11 / `results/round11-dirty-*.json`.)*
14. **Round 12 — off the main thread (`worker`, `whybrid`) BREAKS the paint floor. This was the last unlock with breakthrough potential, and it is proven IN as a capability / OUT of the `data-*` model.** Both render to one `OffscreenCanvas` via `transferControlToOffscreen`, zero per-cell DOM. `worker` = a dedicated worker owns produce+diff+paint (main thread idle); `whybrid` = reactive `state()` stays on main, a smart-style loop diffs and posts changed cells (transferred `Float32Array`) to the worker to blit (paint-only off-thread). **`worker` input: 5.8–10.5 ms @3000, 9.2 @8000** — 60–170× below `smartcv` (603–1576) and *below the main-thread `canvas` floor* (32 ms); main thread 77–93% free. **`whybrid` input: 21.5–47.7 ms** — 15–33× over `smartcv` while KEEPING main-thread state. Decisive qualitative finding: **input decouples from render load** — with the worker software-throttled 20× (slow-device parity), input still holds ~5–7 ms while worker pixel-fps degrades gracefully (40→22→7→2.4 across dashboard-3k→dash-8k→storm-3k→storm-8k); slowness never touches input because paint no longer shares a thread with it. **Verdict: the only lever that beats the DOM paint floor, but it wins by leaving the DOM** (no per-cell elements → no `data-*`/events/CSS/a11y). Does NOT enter the core; belongs (if the product wants it) as a separate opt-in **canvas-renderer addon** for canvas-shaped content (numeric grids, heatmaps, dashboards). `whybrid` proves the reactive half integrates with no kernel change, so it's buildable today; building it is a product-scope call, not a research one. *(FINDINGS Round 12 / `results/round12-headline.json`, `round12-worker-throttled.json`, `round12-scale8000.json`.)* **Critical measurement note: CDP `Emulation.setCPUThrottlingRate` does NOT reach a dedicated worker** (playwright-core here can't attach an `Emulation` session to a worker target; the worker's own busy-probe reads ~1× under the page's 20×). A bare `worker` run is therefore a "free fast core" analogue; `--wthrottle`/`?wrate=` applies a software per-frame stretch in the worker for the honest slow-device number (both reported).
15. **Round 13 — cheaper per-cell DOM paint: `backgroundColor` vs `--h` (no win), containment (a real but bounded win), fixed size (no win).** Three URL knobs (`paint=var|bg`, `contain=off|content|strict`, `fixed=1`), scheduler held constant across `off`/`cv`/`smartcv`, dashboard 3000, text=0 (paint isolated). **(a) `style.backgroundColor` is NOT cheaper than the inheriting `--h` custom property** — the open question is answered *no*: a cell has no inheriting subtree, so `--h` invalidates only that element, same scope as `backgroundColor`; `smartcv` var 385 vs bg 398 (wash), `cv` var 499 vs bg 542 *worse-despite-lighter-throttle*. **(b) CSS containment is a real lever but only for the un-culled `off` path**: `off`+`contain:strict` 1671→**483 ms (~3.5×, at harsher throttle so understated)** by making each cell an independent layout/paint root — but `content-visibility` already *implies* containment, so `cv`/`smartcv` gain nothing (499→528, 385→463). At 3000, `off`+contain (483) ≈ `cv` (499): containment alone captures most of cv's win at this scale; cv's extra margin is offscreen culling (scales with grid≫viewport). Guidance: use `contain: content` on cells if you can't use `content-visibility`; redundant if you can. **(c) Fixed-size cells: no win** — the grid already sizes cells deterministically (`grid-auto-rows` + `1fr`), nothing to remove. Best DOM config here (~385–500) still 24× above `canvas` (16) and 40–80× above `worker` (5.8): refines the decomposition, does not move the floor. Throttle drifted 11–29× across the batch — conclusions rest on drift-surviving directions. *(FINDINGS Round 13 / `results/round13-t0-*.json`, `round13-t1-*.json`.)*
16. **Round 14 — scroll-driven occlusion closes AGAINST `vis`; occlusion is dead in every regime.** `run.mjs` gained `--scroll` (oscillates a mouse wheel through the grid all measure window; amplitude clamped to the page's real overflow). `cv`/`smartcv`/`vis` at dashboard 12000, ±scroll. **Under scroll (all clean ~23×): `smartcv` 2695 vs `vis` 4792 — `vis` is +78% WORSE**, exactly in the scenario Round 9 flagged as occlusion's best case. Cause: scrolling churns the `IntersectionObserver` every frame → O(N) `visRebuild()` each change → revealed rows arrive stale so the budget burns on reveal-repaint; that overhead swamps the sweep-culling. `vis` also reveals *staler* pixels (max staleness 26 s vs `smartcv` 19 s). `smartcv` stays the best DOM config under scroll (beats plain `cv` 5638 too — content-visibility pays a synchronous reveal cost). Scrolling ≈ doubles effective render cost but stays paint-bound (nothing near the `worker` 5.8 ms floor). **Verdict: occlusion write-culling is null static (R9) and harmful scrolling (R14) — it culls the cheap sweep, never the paint. `content-visibility` (culls paint, already in `smartcv`) is the only box-count lever worth using.** `mode=vis` is a fully-settled negative. *(FINDINGS Round 14 / `results/round14-scroll.json`, `round14-noscroll.json`.)*

## Measurement gotchas (learned the hard way — respect these)

- **`worstBlock`/Long-Tasks under CPU throttle is inflated** by throttle-inserted pauses (wall-clock ≠ CPU time). Use it only for *relative* comparisons under the same throttle. **Input-delay** (`performance.now() - event.timeStamp` in the handler) is the clean absolute metric.
- **The jank-dot animation is a huge confound** — it forces per-frame repaints scaling with DOM size (~4.7fps quiet at 3000 with dot vs 24.7 without). Always `dot=0`.
- **`page.keyboard.type` deadlocks** when a frame is multi-second (Playwright awaits the renderer per key). Inject keys fire-and-forget via CDP `Input.dispatchKeyEvent` instead (run.mjs does this).
- **Do NOT edit `main.js` while a run is in flight** — Vite HMR reloads the page and kills the run ("Execution context destroyed by navigation"). Edit between runs.
- **Do NOT `pkill -f chrome` from the Bash tool** — it destabilizes the sandbox shell. Let `runCase` close its own browser (it does, via try/finally).
- **The dev server dies periodically** (container churn / HMR). Re-`curl` it before each batch and restart if `000`.
- **CDP throttle drifts 12–36×** between single cases; trust multi-case sweep medians, not single A/Bs, and re-probe every case (drivers do).
- **The O(N) `currentBacklog` poll** used for convergence saturates the throttled thread — convergence numbers for pull/smart/sliced are unreliable; input numbers are fine.

## What has NOT been tried yet (next-session candidates)

1. ✅ **DONE (Round 6) — Kernel no-subscriber fast-path.** Applied in `src/core/state.js` (`// EXPERIMENT:`). Not just `notifySubscribers` — the whole `set`-trap notify/flush path is skipped when a store has no listeners + no beforeFlush hooks. `pull` storm write-pass frame 156→16ms; kernel write path ~9%→~1.5%; 488 tests pass. Next step for *this* one is graduation to `main` with the full artifact matrix, not more measurement. *(FINDINGS Round 6.)*
2. ✅ **DONE (Round 6) — Pull renderer over PLAIN data.** Added as `mode=plain`. Profile confirms ~0% kernel write path (the theoretical floor). Finding: after the fast-path (item 1) the reactive `pull` write pass is already within ~1–2× of plain, so bypassing `state()` buys only the last sliver — not worth losing reactivity for the churny layer. *(FINDINGS Round 6.)*
3. ✅ **DONE (Round 7) — `smart` + `cv` combined (`mode=smartcv`).** The layers compose: dashboard 3000, median input `off` 1887 → `smart` 1643 → `cv` 662 → **`smartcv` 515 ms** (fps 0.3→1.0), best of all four. Paint (not scheduling) is the bottleneck, so `cv` (paint culling) is the dominant lever and `smart` is a secondary multiplier that only surfaces once `cv` uncovers it; at 8000 cells `smart` *alone* is even worse than `off` (pull sweep overhead), while `smartcv` is ~6× better than `off`. *(FINDINGS Round 7 / `results/exp3-smartcv.json`.)*
4. **Recency-priority within the slice** (paint most-recently-changed first, with a fairness floor) — the "render the newest batch, drop older" half of principle 1 at sub-frame granularity. `bounded`/`pull` use index-order cursors; recency ordering is untested.
5. **Adaptive budget by measured sustainable rate** (grow/shrink the budget from recent frame timing), vs the current fixed 2ms/10ms input-reserved toggle. The "continuously measure how much we can sustain" idea is only crudely approximated.
6. **A clean freshness/convergence measurement** (fix the O(N) poll: maintain an O(1) backlog counter, or sample the drain curve at fixed times instead of polling to 0) to finally show `smart`'s "best of both" on the freshness axis. *(Partially unblocked: `mode=dirty` now exposes an O(1) `dirtySink.size` backlog — Round 11. Measure convergence on `dirty`, or port its O(1) counter to `smart`.)*
7. **Real hardware / lighter throttle** (4–8×) to separate the genuine signal from throttle wall-clock inflation, and to test the "fast machine → bigger renderQueue win" caveat.
8. ✅ **DONE (Round 12) — Web Worker + OffscreenCanvas.** Built as `mode=worker` (full off-thread produce+diff+paint) and `mode=whybrid` (main-thread `state()` + diff, paint-only off-thread). **BREAKS the paint floor** — `worker` input 5.8–10.5 ms (vs `smartcv` 603–1576), below even the main-thread `canvas` floor; input decouples from render load (holds ~5–7 ms even with the worker throttled 20×). `whybrid` keeps main-thread reactivity and is still 15–33× over `smartcv`. **Verdict: the only lever that beats the DOM floor, but wins by leaving the DOM** — ships (if at all) as a separate opt-in canvas-renderer addon, not a core/handler change. *(FINDINGS Round 12.)*
9. ✅ **DONE (Rounds 9 + 14) — Occlusion-driven write culling (`mode=vis`) — dead in every regime.** Built as `smartcv` + an `IntersectionObserver` visible-set (sweep/writes only visible; offscreen refresh on reveal). **NULL static (Round 9)** — at 12000 clean, `smartcv` 1847→2465 vs `vis` 2508→2419, indistinguishable — because it culls the cheap sweep, not the paint. Round 14 tested its one remaining hope, **scrolling** (`run.mjs --scroll`), and it came out **+78% WORSE than `smartcv`** (4792 vs 2695 @ dashboard 12000, ~23×): scrolling churns the observer every frame → O(N) list rebuilds + stale reveal-repaint, overhead that swamps the sweep-culling. Occlusion is settled negative on both axes; only `content-visibility` (culls paint) is worth using. *(FINDINGS Rounds 9, 14.)*
10. ✅ **DONE (Round 11) — O(changed) dirty-list pull (`mode=dirty`).** Built the cheapest change-signal: a kernel `__setDirtySink(set)` that adds each changed store to a sink inside the fast path (O(1), no notify; 488 tests pass). `dirty` drains it O(changed) instead of sweeping O(N). **No consistent input win** (dashboard 3000 tie, 12000 clean worse, storm better — all inside throttle noise): input is paint-bound (Round 8) and the sweep was never O(N)/frame (budget breaks it early). The measured "tension" the item predicted is real: `sink.add` **taxes the write path** (`prod` 61 vs 42 at 12000), partly undoing the Round-6 fast-path. Sole clean win is an **O(1) backlog** (measurement aid). Recency-ordering (item 4) was NOT pursued — with no input headroom to chase, ordering the drain newest-first has no lever to pull. A flat dirty-*array* (vs the `Set` used) might beat the sweep on the write pass, but can't change the paint-bound verdict. *(FINDINGS Round 11.)*
11. **`isInputPending()` / `scheduler.yield()` cooperative budget.** Replace the fixed 2ms/10ms input-reserved toggle in `smart` with `navigator.scheduling.isInputPending()` checks inside the slice loop (yield the *instant* input is queued, use the whole frame otherwise). Chrome-only but feature-detectable progressive enhancement; standards-track, directly targets the headline metric (input delay). Candidate `mode=coop`. Also the natural place to test item 5's measured-rate adaptive budget.
12. ✅ **DONE (Round 8) — Paint-floor probe (`mode=canvas`).** Reactive stores + byte-identical `pull` schedule, painting changed cells into one `<canvas>` (grid hidden) instead of N DOM boxes. Result: **median input 16 ms / 12.8 fps** vs `smartcv` 388 / 1.3 (24×) and `off` 1977 / 0.3 (89×), dashboard 3000/20×. Decomposition: `pull`≈`plain` (write ~3% of input), `pull`→`canvas` (DOM paint ~99%). **Verdict: scheduling headroom is exhausted** — `smartcv` is within a small constant of the best any scheduler can do in the per-cell DOM model; the only large levers left reduce the live DOM-box count (item 9 / virtualization) or leave per-cell DOM (canvas, out of scope). One-line follow-up still open: canvas with a *larger* budget (cheap paint has budget headroom DOM lacks) to show it can be both fresh AND responsive. *(FINDINGS Round 8.)*
13. **Productize the winning stack** — 9–12 have now settled, so the decision is clearer. **Ships: the Round-6 fast-path graduates to `main`** (settled, artifact-matrix work). The `smartcv` recipe becomes a **docs guide** ("high-churn rendering": `content-visibility` + a drop-stale input-reserved pull renderer over unsubscribed stores + colour-only cells) rather than a kernel feature — because Round 11 showed the `$version`/dirty-sink does *not* make a generic addon meaningfully faster (input is paint-bound), so there's no near-zero-cost kernel primitive worth adding for it. `__setDirtySink` should be **dropped** when the fast-path graduates (it's a measurement aid, not a lever). A thin `lume-js/addons` pull-renderer is still viable but its value is ergonomics, not the speed the sink was meant to buy.

## Verdict as it stands

KILL `renderQueue` as designed (freeze + starvation + dominated + over budget). The constructive direction, layered by cost: **(1) reduce render work** — `content-visibility` (only thing that raises the fps ceiling); **(2) drop-stale, cursor-fair, input-reserved pull rendering** — `smart`/`pull`/`bounded`; **(3) the write-pass floor** — *now addressed* by the Round-6 kernel no-subscriber fast-path (`src/core/state.js`), which makes an unsubscribed store's write ~10× cheaper (156→16ms/frame) and lands it within ~1–2× of plain data, so the churny layer no longer needs to abandon `state()`. Layer 3's remaining O(N) is the irreducible `Proxy` trap + DOM paint, not the notify/flush machinery.

**The constructive stack is now empirically complete and ordered (Round 7):** Layer 1 `content-visibility` is load-bearing (only lever that cuts the dominant cost, paint; free CSS), Layer 2 `smart` (input-reserved drop-stale pull) is a secondary multiplier that only surfaces once Layer 1 uncovers it — and can *hurt* at large grids if used alone — and Layer 3 (the write floor) is gone via the Round-6 fast-path. Best configuration measured end-to-end: **`smartcv`** (content-visibility + smart pull over fast-path'd unsubscribed stores), dashboard 3000 median input 515 ms vs `off` 1887 ms, fps 1.0 vs 0.3.

**Rounds 8–11 closed the scheduling frontier (this session).** The paint-floor probe (Round 8, `canvas`) proved DOM-per-cell paint is **~99 %** of what's left: median input 16 ms with the same reactive reads + schedule but no per-cell DOM, vs `smartcv`'s ~400 ms (24×). That prediction — "the schedule is spent; only reducing DOM-box work can help" — then held for **both** remaining scheduling ideas: **occlusion write-culling (Round 9, `vis`)** is null on input (it culls the cheap sweep, not the paint), and **O(changed) dirty pull (Round 11, `dirty`)** is null on input and even taxes the write path (`sink.add`), with a `Set` losing to the plain array sweep. The one genuinely new *work-reduction* lever found is **per-cell text/layout (Round 10)**: dropping `textContent` (colour-only cells) cuts `cv` ~51 % / `smartcv` ~23 % — shippable guidance, though still 19× above the canvas floor, so the DOM box itself remains the residual.

**Rounds 12–14 closed the study (this session).** Round 12 took the off-main-thread bet (`worker`/`whybrid`) and it is **the only lever that beats the paint floor** — input 5.8–10.5 ms, decoupled from render load — but it wins by leaving the DOM, so it is a canvas-addon question, not a `data-*` core change. Rounds 13–14 then swept the remaining DOM-side levers to exhaustion: `backgroundColor`-vs-`--h` and fixed-size cells are non-levers (R13), CSS containment is a real but bounded win already subsumed by `content-visibility` (R13), and occlusion write-culling (`vis`) is null static and *harmful* under scroll (R14). Nothing DOM-side reaches the canvas/worker floor. The constructive frontier is now closed on both axes: schedule (Rounds 1–11) and per-cell DOM paint (Rounds 13–14) are spent; the only thing past the floor is leaving per-cell DOM (Round 12).

**Highest-leverage next steps (re-ordered by this session's evidence):**
- (a) **Graduate the Round-6 fast-path to `main`** — behaviourally transparent (488 tests green), an artifact-matrix exercise, not more research. Drop `__setDirtySink` (Round 11: not a lever).
- (b) **Write the "high-churn rendering" docs guide** — `content-visibility` + drop-stale input-reserved pull + **colour-only cells (Round 10)**. This is the product of the whole study; `smartcv`+text is the recipe.
- (c) The research directions with headroom left split by *which* cost they attack, and it's worth being precise because Round 8's decomposition sharpens it. The per-frame input floor under churn is the paint of the **(visible ∩ changed)** DOM cells. **Box-count reduction** — virtualization, occlusion, `content-visibility` — only removes *offscreen* boxes, so it helps when the grid ≫ viewport but does **not** reduce the visible-changed paint that bounds input when the grid fits the viewport; virtualization approaches `cv`, **not** the canvas floor (a common trap — don't expect it to). The lever that lowers the *visible* floor within `data-*` is **cheaper per-cell paint**: Round 10 did the text/layout half (−23–51 %); Round 13 did the rest — `backgroundColor` vs `--h` is a *non*-lever (answered), CSS **containment** is a real ~3.5× win for the un-culled `off` path but redundant once you use `content-visibility`, and fixed-size cells are a non-lever. So the DOM-paint levers are now exhausted, and none reach the canvas/worker floor. Still open (Round 14, below): **scroll-driven occlusion** (Round 9's untested half — `vis` under an actually-scrolling driver, occlusion's real use case, needs driver scroll support).
- (d) Lower-leverage refinements still valid: item 11 (`isInputPending()`/`scheduler.yield()` cooperative budget), items 4–8. But note Round 8's caveat: any input-side scheduling refinement is polishing the ~1 % that isn't paint.
- (e) ✅ **Off-main-thread (Round 12) — RESOLVED.** Worker + OffscreenCanvas (`worker`/`whybrid`) is the only lever that beats the paint floor: input 5.8–10.5 ms and decoupled from render load. But it wins by leaving the DOM, so it is **not** a `data-*` core/handler change — it is a candidate for a *separate opt-in canvas-renderer addon* for canvas-shaped content, and building it is a product-scope decision, not open research. This was the last direction with breakthrough potential; with it answered, the study's constructive frontier is closed (see the final verdict below).
