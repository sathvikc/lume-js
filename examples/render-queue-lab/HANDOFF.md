## The question

Can Lume keep the UI responsive (input smooth, pixels current) when thousands of reactive cells update per frame under heavy CPU load (DevTools 20├ù throttle)? The frozen `feature/render-queue` addon (`renderQueue`) was the first attempt; the maintainer wants the whole direction stress-tested and, where possible, improved or reinvented.

## What has been TRIED and FOUND (do not redo)

1. **The freeze is real and root-caused.** `renderQueue` drains ~16 cells/frame (2ms budget under 20├ù) while a storm re-dirties thousands; backlog pins, staleness grows unbounded. Budget tuning (2ΓåÆ32ms) does not fix it. *(FINDINGS "The freeze".)*
2. **Starvation bug in `renderQueue`.** Under uniform re-dirty order (a grid iterating cells 0..N ΓÇö the common case) the Set drain re-adds every entry in the same order, so it repaints ~8 front cells up to 13├ù while **2984/3000 never paint**. The addon's no-starvation test only re-dirties one entry, so it ships green. `bounded`/`pull`/`simple` do not starve. *(FINDINGS Round 2; `round2.mjs starve`.)*
3. **renderQueue's input win is ~1.5├ù at best (median), never ΓëÑ5├ù**, and it freezes pixels under sustained load. `content-visibility` (free CSS) beats it 2ΓÇô3├ù in realistic regimes. **Verdict: KILL** the shipped addon. Also 0.37 KB over the 8 KB all-in-one budget (budget kept at 8; the silent 8ΓåÆ9 raise was reverted). *(FINDINGS Rounds 1ΓÇô2.)*
4. **Prototype's 7.4├ù vs my ~1.5├ù** = fixed-floor : spreadable-apply ratio. Headed (xvfb) did NOT change it ΓÇö floor persists in both paths, so it is not a headless artifact. Residual caveat: both environments are ~5├ù slower natively than the prototype's machine, so on fast hardware the input benefit could be larger. Freeze/starvation are hardware-independent. *(FINDINGS Round 2/3; `round2.mjs fidelity`.)*
5. **fps ceiling.** Native, a full storm holds 60fps for only a few hundred cells; a dashboard ~1200. Rescheduling (pull/bounded/rq) does NOT raise the ceiling (same total work). Only **reducing work** raises it: `content-visibility` holds 60fps to ~2000 dashboard cells. *(FINDINGS Round 3; `headroom.mjs`.)*
6. **Main-thread free time.** Budgeted schedulers (pull/bounded) roughly HALVE the worst blocking task vs `off` ΓÇö that (not fps) is what they buy. But nothing frees the thread from the WRITE pass. *(FINDINGS Round 3.)*
7. **`smart` (drop-stale + input-reserved budget)** had the best input of the pull family (burst: 1218ms vs 1486ΓÇô1557ms fixed). Principle 2 works, but its ceiling is the write pass. *(FINDINGS Round 4; `smart.mjs`.)*
8. **Layer 3 (`sliced`): time-slicing the write pass.** Chunking WORKS at the JS level (producer 7ms, slice 16ms, budget-respected). But a CPU profile shows the real floor is the **Lume kernel write path** ΓÇö `set` + `notifySubscribers` + `flushBatchedStates` Γëê **27% of active CPU**, O(N) in writes, even with ZERO subscribers. Chunking spreads it, doesn't reduce it; input at 3000 cells (~1155ms) Γëê pull/off. *(FINDINGS Round 5 / `results/layer3-summary.json`.)*
9. **Round 6 ΓÇö the write-path floor is removed, two ways.** (a) **Kernel no-subscriber fast-path** (`src/core/state.js`, applied): the `set` trap skips notify/flush when a store has no listeners + no beforeFlush hooks. `pull` storm worst write-pass frame **156ΓåÆ16ms (~10├ù)**; profile kernel write path **~9%ΓåÆ~1.5%** (`notifySubscribers`/`takeEffects` vanish); **all 488 tests pass unchanged**. (b) **`plain` mode** (pull over a plain array, no `state()`): profile shows **~0% kernel write path** (97% `(program)`), `prod` ~8ΓÇô16ms. The fast-path captures ~85% of the win *while keeping the reactive API*; plain buys only the last sliver by dropping reactivity. Neither raises the fps ceiling (still content-visibility) nor fixes input under a pure paint-bound storm ΓÇö they remove the O(N) write floor Round 5 diagnosed. Use `prod` (worst write-pass frame), not input-delay (paint-dominated) or `worstBlock` (throttle-inflated). *(FINDINGS Round 6 / `results/round6-writepath-floor.json`, `exp1-fastpath.json`, `exp2-plain.json`.)*
10. **Round 8 ΓÇö the paint floor is 16 ms; DOM-per-cell is ~99% of what's left, so scheduling headroom is exhausted.** The `canvas` probe (reactive stores + byte-identical `pull` schedule, but painting into one `<canvas>` instead of N DOM boxes) hits **median input 16 ms / 12.8 fps** at dashboard 3000/20├ù ΓÇö vs `smartcv` **388 ms / 1.3 fps** (24├ù gap) and `off` **1977 ms / 0.3 fps** (89├ù). Decomposition in the same throttle band: `pull` (1420) Γëê `plain` (1378) ΓåÆ the reactive **write** is now ~3% of input (fast-path confirmed on the input axis); `pull` (1420) ΓåÆ `canvas` (16) ΓåÆ the **DOM paint** of individual boxes is the other ~99%. Consequence for the whole study: every scheduler moves *when* the N DOM writes happen but not *that* there are N of them, and Round 8 shows those writes are the cost ΓÇö so `smartcv` is within a small constant of the best any scheduler can do inside the per-cell DOM model. **The only remaining large levers reduce the live DOM-box count** (occlusion write-culling ΓåÆ item 9; virtualization) or leave per-cell DOM (`canvas`/`OffscreenCanvas`, out of scope). Stop optimizing the schedule; reduce the boxes. *(FINDINGS Round 8 / `results/round8-item12-paintfloor.json`.)*
11. **Round 9 ΓÇö occlusion write-culling (`vis`) is a NULL result on input; confirms Round 8.** `vis` = `smartcv` + an `IntersectionObserver` visible-set (sweep/writes touch only visible cells). Occlusion verifiably active (`visVisibleCount` 3556/8000, 4340/12000). But at dashboard 12000, two clean ~22ΓÇô23├ù runs give `smartcv` 1847ΓåÆ2465 and `vis` 2508ΓåÆ2419 ΓÇö **the direction flips, so input is indistinguishable**. The sweep it culls was never the bottleneck (writes ~3% of input, Round 8); the visible paint it can't cull is. `vis` even paints *more* visible boxes/frame (budget concentrated on-screen), a mild mechanism against it. Only genuine move: visible backlog lower (920ΓÇô1365 vs 3061ΓÇô3256) but denominator-confounded and absent from input/staleness. Occlusion does NOT raise the ceiling like `cv` did (cv culls paint, vis culls only the sweep). Its real use case ΓÇö **scrolling** ΓÇö is untested (driver never scrolls). *(FINDINGS Round 9 / `results/round9-vis-confirm.json`.)*
12. **Round 10 ΓÇö per-cell text (layout) is a real SECONDARY lever, ~a quarter to a half of DOM cost.** Same modes with `text=1` (number+colour) vs `text=0` (colour only), dashboard 3000/20├ù: `off` 1935ΓåÆ1569 (ΓêÆ19%), `cv` 889ΓåÆ437 (**ΓêÆ51%**), `smartcv` 411ΓåÆ318 (ΓêÆ23%), `canvas` 26ΓåÆ17 (ΓêÆ34%) ΓÇö and every `text=0` case ran at equal-or-harsher throttle, so the savings are *understated*. Dropping per-cell `textContent` (which forces layout/reflow) is shippable guidance: colour-only cells (CSS var/class, keep the number out of the churny path) are materially cheaper. **But text is not the floor** ΓÇö even colour-only, `smartcv` (318) is still 19├ù above `canvas` (17); the dominant residual is the per-cell style-recalc + background paint of the DOM box itself, which only `canvas` removes. Untested adjacent lever: direct `style.backgroundColor` vs the inheriting `--h` custom property. *(FINDINGS Round 10 / `results/round10-text{0,1}.json`.)*
13. **Round 11 ΓÇö O(changed) dirty pull (`dirty`) is NOT a responsiveness lever; confirms Round 8 a third time.** New kernel `EXPERIMENT`: opt-in `__setDirtySink(set)` adds each changed store to a sink inside the fast path (O(1), no notify ΓÇö 488 tests still pass). `mode=dirty` = `smartcv` but drains the sink O(changed) instead of sweeping O(N). Result: **no consistent input win** ΓÇö dashboard 3000 indistinguishable (449 vs 483), dashboard 12000 clean *worse* (1824 vs 2464), storm 3000 better-despite-harsher-throttle (458@17.9├ù vs 360@23.6├ù): all inside the throttle-drift noise. Input is paint-bound, and the sweep was never O(N)-per-frame anyway (budget breaks it after ~paints/churn scans). Two findings cut *against* it: the `sink.add` **taxes the write path** (`prod` 61 vs 42 at 12000, partly undoing the Round-6 fast-path win), and a `Set` is the wrong structure (its churn loses to the sweep's monomorphic array reads; a flat dirty-array would close the constant gap but not the structural verdict). Sole clean win: **O(1) backlog** (`dirtySink.size`) ΓÇö a measurement aid for future convergence curves, not a perf lever. *(FINDINGS Round 11 / `results/round11-dirty-*.json`.)*

## Measurement gotchas (learned the hard way ΓÇö respect these)

- **`worstBlock`/Long-Tasks under CPU throttle is inflated** by throttle-inserted pauses (wall-clock Γëá CPU time). Use it only for *relative* comparisons under the same throttle. **Input-delay** (`performance.now() - event.timeStamp` in the handler) is the clean absolute metric.
- **The jank-dot animation is a huge confound** ΓÇö it forces per-frame repaints scaling with DOM size (~4.7fps quiet at 3000 with dot vs 24.7 without). Always `dot=0`.
- **`page.keyboard.type` deadlocks** when a frame is multi-second (Playwright awaits the renderer per key). Inject keys fire-and-forget via CDP `Input.dispatchKeyEvent` instead (run.mjs does this).
- **Do NOT edit `main.js` while a run is in flight** ΓÇö Vite HMR reloads the page and kills the run ("Execution context destroyed by navigation"). Edit between runs.
- **Do NOT `pkill -f chrome` from the Bash tool** ΓÇö it destabilizes the sandbox shell. Let `runCase` close its own browser (it does, via try/finally).
- **The dev server dies periodically** (container churn / HMR). Re-`curl` it before each batch and restart if `000`.
- **CDP throttle drifts 12ΓÇô36├ù** between single cases; trust multi-case sweep medians, not single A/Bs, and re-probe every case (drivers do).
- **The O(N) `currentBacklog` poll** used for convergence saturates the throttled thread ΓÇö convergence numbers for pull/smart/sliced are unreliable; input numbers are fine.

## Verdict as it stands

KILL `renderQueue` as designed (freeze + starvation + dominated + over budget). The constructive direction, layered by cost: **(1) reduce render work** ΓÇö `content-visibility` (only thing that raises the fps ceiling); **(2) drop-stale, cursor-fair, input-reserved pull rendering** ΓÇö `smart`/`pull`/`bounded`; **(3) the write-pass floor** ΓÇö *now addressed* by the Round-6 kernel no-subscriber fast-path (`src/core/state.js`), which makes an unsubscribed store's write ~10├ù cheaper (156ΓåÆ16ms/frame) and lands it within ~1ΓÇô2├ù of plain data, so the churny layer no longer needs to abandon `state()`. Layer 3's remaining O(N) is the irreducible `Proxy` trap + DOM paint, not the notify/flush machinery.

**The constructive stack is now empirically complete and ordered (Round 7):** Layer 1 `content-visibility` is load-bearing (only lever that cuts the dominant cost, paint; free CSS), Layer 2 `smart` (input-reserved drop-stale pull) is a secondary multiplier that only surfaces once Layer 1 uncovers it ΓÇö and can *hurt* at large grids if used alone ΓÇö and Layer 3 (the write floor) is gone via the Round-6 fast-path. Best configuration measured end-to-end: **`smartcv`** (content-visibility + smart pull over fast-path'd unsubscribed stores), dashboard 3000 median input 515 ms vs `off` 1887 ms, fps 1.0 vs 0.3.

**Rounds 8ΓÇô11 closed the scheduling frontier (this session).** The paint-floor probe (Round 8, `canvas`) proved DOM-per-cell paint is **~99 %** of what's left: median input 16 ms with the same reactive reads + schedule but no per-cell DOM, vs `smartcv`'s ~400 ms (24├ù). That prediction ΓÇö "the schedule is spent; only reducing DOM-box work can help" ΓÇö then held for **both** remaining scheduling ideas: **occlusion write-culling (Round 9, `vis`)** is null on input (it culls the cheap sweep, not the paint), and **O(changed) dirty pull (Round 11, `dirty`)** is null on input and even taxes the write path (`sink.add`), with a `Set` losing to the plain array sweep. The one genuinely new *work-reduction* lever found is **per-cell text/layout (Round 10)**: dropping `textContent` (colour-only cells) cuts `cv` ~51 % / `smartcv` ~23 % ΓÇö shippable guidance, though still 19├ù above the canvas floor, so the DOM box itself remains the residual.
