What renderQueue changes (and what it never touches)

Without priority scheduling, each frame is one atomic task: write all stores, run all cell effects, restyle every cell. Under a throttled CPU that task runs hundreds of milliseconds, and every keystroke waits for all of it. With priority scheduling, cell effects split into a tracked read (runs in the flush, marks the cell dirty, costs almost nothing) and a DOM write (runs from a queue with a fixed per-frame budget, visible cells first, stopping early when isInputPending() reports a waiting user). 

queue.schedule({
  el: cell,
  track: () => s.value,                       // flush-time: tracked, cheap
  apply: () => cell.style.setProperty(...),   // drained within the frame budget
});

Truth is never delayed: the stores, the aggregate, and anything bound to them stay synchronous. Only pixels lag, by a measured, displayed amount.
