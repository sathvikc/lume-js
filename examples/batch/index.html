A dashboard of 200 independent stores (one per cell), each with its own fine-grained effect, plus one aggregate effect that reads all 200. The benchmark fires 100 write-bursts; every burst mutates all 200 stores. Per-store batching runs the aggregate once per store flush — batch() runs it once per burst. The point

Everything on this page — 200 stores, 201 effects, two-way DOM binding, the batcher — is 2.66 KB gzipped of library, no build step, valid HTML. In component frameworks, a multi-store burst like this re-renders every subscribed component unless you hand-tune memoization and selectors; the framework runtime alone ships 15–45 KB before your first line. Here the entire optimization is one explicit, standard-JavaScript function call: 

batch(() => {
  for (const s of stores) s.value = next(s.value);
});

Fair-play notes: both modes await a timeout between bursts (an event-like cadence), so scheduling overhead is identical; the per-cell effects run the same number of times in both modes; only the cross-store aggregate differs — that is exactly what batch() deduplicates. Without batch, Lume's per-store microtask batching is still doing its normal job — this page measures the one case it deliberately does not cover, and what the opt-in escape hatch buys.
